# Примеры компонентов

Playground и репозиторий примеров полезны как источник разметки, но сами по
себе не заменяют release evidence. У playground может быть собственная
локальная конфигурация ассетов, поэтому пример становится документационным
примером только после проверки на lock проекта.

## Как оформить проверенный пример

У примера в справочнике должны быть указаны:

- идентификатор правила компонента;
- путь к исходному примеру;
- Core/Smart lock, на котором он проверен;
- ожидаемый визуальный и интерактивный результат;
- проверка клавиатуры, фокуса и ошибок сети;
- дата или revision проверки.

Если название группы примеров и название runtime-правила различаются, добавьте
явное соответствие в каталог. Не выводите его из сходства имён.

## Рекомендуемый порядок работы

1. Найдите элемент в [каталоге](/ru/components/catalog/).
2. Откройте связанный исходный пример и оставьте только минимальную разметку
   для нужного сценария.
3. Подключите его через [контракт поставки](/ru/components/connection/).
4. Проверьте страницу в браузере и только после этого переносите фрагмент в
   проектную документацию или шаблон.

## Что проверять дополнительно

Для формы или контрола проверьте подпись, состояние disabled, фокус и работу с
клавиатуры. Для overlay-элемента — открытие, закрытие и возврат фокуса. Для
компонента со списком или данными — пустое состояние и ошибки. Если эти задачи
выходят за локальное поведение компонента, используйте соответствующий раздел
[Smart Components](/ru/smart-components/introduction/).
