# Загрузка и зависимости Smart Components

Для каждой reference-страницы Smart Components должна быть подтверждённая
карту жизненного цикла: входную разметку или конфигурацию, ассеты, зависимости
и наблюдаемый результат. Эту карту нельзя выводить только из имени файла или
похожего примера.

## Базовая последовательность

1. Утверждённая схема фиксирует входную разметку или конфигурацию сценария.
2. Source evidence фиксирует контракт элемента и необходимые зависимости.
3. Lock определяет поставляемые ассеты, которые допустимо использовать в этом
   сценарии.
4. Браузерная проверка подтверждает, что зависимости доступны и сценарий
   достигает ожидаемого результата.
5. События, данные и очистка состояния описываются только там, где они
   подтверждены для конкретной записи.

## Почему нельзя описывать Smart Components одной общей схемой

Набор зависимостей, собственных стилей, разметки и событий различается у
компонентов. Один элемент может опираться на обычный component-ассет, другой —
на отдельный контракт Smart Components, а третий — на оба слоя. Поэтому в
каталоге фиксируются подтверждённые связи и фактические ассеты для каждой
записи.

Не переносите в общий раздел предположения о кэше, серверных шаблонах,
формате данных или событиях. Эти детали публикуются только там, где они
подтверждены исходниками и проверены на release lock.

## Что проверять при изменении зависимости

- запросы к ассетам и отсутствие `404`;
- порядок инициализации связанных элементов;
- повторную загрузку и очистку состояния;
- клавиатурное управление и возврат фокуса для интерактивных элементов;
- пустые данные, ошибку загрузки и отключённое состояние, если они входят в
  контракт конкретного компонента.

Состав каждой записи см. в [каталоге](/ru/smart-components/catalog/). Для
физической поставки файлов и шаблонов используйте страницу
[Шаблоны и ассеты](/ru/smart-components/templates-and-assets/).
